# HW_6_Architecture

# Защищённый финансовый портал – демонстрация мер безопасности



## Обзор

Данный проект реализует три ключевых аспекта безопасности веб-приложения:

1\. Аутентификация через \*\*OAuth2 / OpenID Connect\*\* (Authorization Code Flow).

2\. Шифрование чувствительных данных\*\* (PII) при хранении в базе данных.

3\. Шифрование канала связи\*\* (TLS) с принудительным HTTPS.



Решение написано на \*\*Python + Flask\*\* с использованием библиотек:

\- `authlib` – OAuth2/OIDC клиент.

\- `cryptography.fernet` – симметричное шифрование.

\- `flask-talisman` – настройка HTTPS и security headers.

\- `sqlalchemy` – ORM.



## Выбор технологий и архитектурные решения



### 1. OAuth2 / OIDC (пункт 1)

- **Поток**: Authorization Code Flow – наиболее безопасный для серверных приложений, так как access token не передаётся через браузер пользователя.

- **Провайдер**: можно использовать Google, Keycloak или любой OIDC-совместимый сервер.

- **Верификация JWT**: ID Token проверяется по подписи (RS256) и стандартным полям (iss, aud, exp).

- **Выход**: реализован локальный logout (очистка сессии) и перенаправление на провайдера для глобального выхода (end\_session\_endpoint).



### 2. Шифрование данных в покое (пункт 2)

- **Алгоритм**: Fernet (симметричное шифрование на базе AES-128 в режиме CBC с HMAC-SHA256). Выбран за простоту использования и автоматическое управление ключами в `cryptography`.

- **Хранение ключа**: ключ загружается из переменной окружения `ENCRYPTION\_KEY`. В production рекомендуется использовать KMS (например, HashiCorp Vault или AWS KMS).

- **Модель пользователя**: поля `full\_name` и `address` шифруются перед записью в БД и расшифровываются при чтении через property-методы. В таблице хранятся поля `\_full\_name` и `\_address` в виде base64-строк.



### 3. Шифрование в движении (пункт 3)

- **TLS 1.2+**: Flask-сервер запущен с `ssl\_context`, предоставляющим самоподписанный сертификат для тестов.

- **Принудительный HTTPS**: `Flask-Talisman(force\_https=True)` перенаправляет все HTTP-запросы на HTTPS.

- **Безопасные cookie**: `SESSION\_COOKIE\_SECURE = True` гарантирует, что сессионная cookie передаётся только по HTTPS.

- **Логирование схемы**: middleware логирует `request.scheme` (демонстрация `https`).



## Соответствие принципам PCI DSS (фрагментарно)

Хотя платёжные данные не обрабатываются, приложение работает с персональными данными (PII). Учтены следующие требования:



- **Пункт 3.4** – шифрование данных учётных записей: поля `full\_name` и `address` зашифрованы перед сохранением.

- **Пункт 4.1** – защита при передаче: используется TLS (HTTPS) для всех запросов, запрещены HTTP-соединения.

- **Предотвращение попадания чувствительных данных в логи**:

&nbsp; - В production следует настроить фильтрацию логов, например, через `logging.Filter`, исключающий поля `full\_name`, `address` и тело запросов.

&nbsp; - В текущей реализации логируется только метод, путь и схема запроса – PII не выводятся.
